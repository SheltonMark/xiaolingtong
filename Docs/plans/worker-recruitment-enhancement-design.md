# 临工招工流程完善设计方案

**日期**: 2026-03-12
**状态**: ✅ 已批准
**优先级**: 高 → 中 → 低（渐进式实现）

---

## 概述

基于已实现的核心招工流程（状态机、接受/拒绝、主管选择、出勤确认），本方案进一步完善用户体验和业务流程，分3个阶段逐步实现8个需求。

**核心目标**:
- 防止临工时间冲突报名
- 支持临工取消报名（带惩罚机制）
- 统一临工端和企业端的状态显示
- 优化企业端UI布局
- 完善主管管理功能
- 建立通知提醒系统
- 实现评价系统
- 完善整体流程（纠纷处理、数据分析）

## Phase 1: 高优先级需求（2周）

### 需求1: 时间冲突检查

**问题**: 临工报名多个工作时，如果时间冲突则不允许报名

**实现逻辑**:
```
当临工报名时：
1. 获取该临工所有"已接受"和"已确认"的应用
2. 对每个应用，检查新报名的工作时间是否与其重叠
3. 时间重叠判断：
   - 新工作开始时间 < 已有工作结束时间 AND
   - 新工作结束时间 > 已有工作开始时间
4. 如果冲突，返回错误并显示冲突的工作信息
```

**API变更**:
- `POST /jobs/:jobId/apply` - 添加冲突检查逻辑
- 返回冲突信息：`{ conflictWith: [{ jobTitle, dateRange, workHours }] }`

**前端实现**:
- 报名时显示冲突提示
- 显示已报名的工作列表
- 提示用户取消或修改已报名工作

**测试用例**: 10个

---

### 需求2: 临工端可以取消报名

**问题**: 临工需要能够取消已报名的工作

**惩罚机制**:
```
取消时间点 | 惩罚
-----------|------
报名后 → 工作前24小时 | 无惩罚
工作前24小时 → 工作前12小时 | 扣信用分5分
工作前12小时 → 工作开始 | 扣信用分10分 + 限制报名24小时
工作开始后 | 扣信用分20分 + 限制报名7天
```

**允许取消的状态**:
- pending (待审核) ✅
- accepted (已接受) ✅
- confirmed (已确认) ✅

**不允许取消的状态**:
- working (进行中) ❌
- done (已完成) ❌
- rejected (已拒绝) ❌
- released (已释放) ❌
- cancelled (已取消) ❌

**API新增**:
- `DELETE /applications/:applicationId/cancel` - 取消报名
- 返回惩罚信息：`{ creditDeduction, restrictionDays, message }`

**前端实现**:
- 我的报名页面显示"取消报名"按钮
- 点击后显示确认对话框
- 显示取消时间和可能的惩罚
- 显示已报名的其他工作

**测试用例**: 8个

### 需求3&5: 状态同步和双向匹配

**问题**: 确保临工端和企业端看到的状态一致

**状态映射表**:
```
后端状态 | 临工端显示 | 企业端显示 | 说明
---------|----------|----------|------
pending  | 待确认   | 待审核   | 企业还未审核
accepted | 待确认   | 已接受   | 企业已接受，待临工确认
confirmed| 已入选   | 已确认   | 双方都已确认
working  | 进行中   | 进行中   | 工作正在进行
done     | 已完成   | 已完成   | 工作已完成
rejected | 已拒绝   | 已拒绝   | 企业拒绝（异常显示）
released | 已释放   | 已释放   | 未及时确认（异常显示）
cancelled| 已取消   | 已取消   | 主动取消（异常显示）
```

**前端实现**:
- 临工端：按状态分类显示（待确认、已入选、进行中、已完成），异常状态单独显示
- 企业端：按状态分类显示（待审核、已接受、已确认、进行中、已完成），异常状态单独显示

**API变更**:
- `GET /applications/my-applications` - 返回按状态分类的应用
- `GET /jobs/:jobId/applications` - 返回按状态分类的应用

**测试用例**: 6个

### 需求6: 企业端选择主管并管理考勤

**问题**: 企业确认招聘零工后，从零工中选择一个主管，对临工的考勤和完成度进行管理

**主管选择流程**:
```
企业接受报名后：
1. 进入"人员分配"页面
2. 选择主管的方式：
   ├─ 方式A: 从报名员工中选择
   │  ├─ 显示所有"已接受"状态的报名者
   │  ├─ 筛选条件：信用分≥95，订单数≥10
   │  └─ 选择后状态变为 'confirmed'，标记为主管
   │
   └─ 方式B: 自己指定管理员
      ├─ 输入管理员信息（姓名、电话、身份证等）
      ├─ 可以是企业自己的员工
      └─ 创建临时管理员账户或关联现有账户
```

**主管权限**:
```
1. 考勤管理
   - 查看所有临工的签到情况
   - 记录临工签到/未签到
   - 确认开工人数

2. 工时/计件记录
   - 记录每个临工的工时
   - 记录每个临工的计件数
   - 上传现场照片

3. 异常上报
   - 上报临工旷工
   - 上报临工早退
   - 上报工时造假
   - 上报其他异常

4. 结算确认
   - 查看工时/计件汇总
   - 确认结算单
   - 提交给企业支付
```

**主管收益**:
```
主管收益 = 正常工资 + 监管服务费

监管服务费 = 监管人数 × 单价 × 工时/件数

示例:
- 监管10人，每人8小时，单价5元/小时
- 监管服务费 = 10 × 5 × 8 = 400元
```

**API新增**:
- `POST /jobs/:jobId/select-supervisor` - 从报名者中选择
- `POST /jobs/:jobId/assign-manager` - 自己指定管理员
- `POST /jobs/:jobId/attendance/checkin` - 签到记录
- `POST /jobs/:jobId/worktime/record` - 工时记录
- `POST /jobs/:jobId/exception/report` - 异常上报
- `GET /jobs/:jobId/settlement/summary` - 结算汇总

**前端实现**:
- 人员分配页面显示两种选择方式
- 考勤管理页面显示签到列表和记录表单
- 工时记录页面显示工时/计件表单
- 异常上报页面显示异常类型和描述

**测试用例**: 10个

### 需求8: 通知提醒系统

**问题**: 关键节点缺少提醒通知

**关键节点通知**:

**临工端通知**:
```
1. 报名成功 → "您已报名，请等待企业审核"
2. 企业接受 → "企业已接受您的报名，请确认出勤"
3. 企业拒绝 → "企业拒绝了您的报名"
4. 出勤提醒 → "工作即将开始，请准时出勤"（工作前1小时）
5. 工作开始 → "工作已开始，请签到"
6. 结算完成 → "工资已到账，请查看"
7. 评价提醒 → "请评价本次工作体验"（工作完成后）
```

**企业端通知**:
```
1. 新报名 → "有新的报名申请，请审核"
2. 临工确认 → "临工已确认出勤"
3. 临工取消 → "临工已取消报名"
4. 工作开始 → "工作已开始"
5. 结算提醒 → "请确认结算单并支付"
6. 临工评价 → "临工已评价您的企业"
```

**通知渠道**:
- 推送消息（小程序）
- 站内信（消息中心）
- 可选：短信/邮件（重要通知）

**API新增**:
- `POST /notifications/send` - 发送通知
- `GET /notifications/list` - 获取通知列表
- `POST /notifications/:id/read` - 标记已读

**前端实现**:
- 消息中心显示所有通知
- 支持按类型筛选
- 支持标记已读/删除

## Phase 3: 低优先级需求（2周）

### 需求7: 零工完成任务后可以对企业打分

**问题**: 零工完成任务后，可以对企业进行打分和评价

**评价流程**:
```
工作完成后：
1. 结算完成 → 推送"请评价企业"通知
2. 临工点击评价 → 进入评价表单
3. 填写评价内容：
   ├─ 星级评分（1-5星）
   ├─ 评价标签（多选）
   ├─ 评价文字（可选）
   └─ 上传照片（可选）
4. 提交评价 → 保存到数据库
5. 企业可见评价 → 显示在企业详情页
```

**评价标签**:
```
正面标签：
- 工作环境好
- 工资按时发放
- 主管很负责
- 工作轻松
- 同事友好

负面标签：
- 工作环境差
- 工资拖欠
- 主管不负责
- 工作强度大
- 同事不友好
- 存在安全隐患
```

**数据模型**:
```
ApplicationRating:
- id
- applicationId
- workerId
- jobId
- rating (1-5)
- content (文字评价)
- tags (标签数组)
- photos (照片URL数组)
- createdAt
- updatedAt
```

**API新增**:
- `POST /applications/:applicationId/rate` - 提交评价
- `GET /jobs/:jobId/ratings` - 查看企业评价列表
- `GET /jobs/:jobId/ratings/summary` - 评价统计（平均分、标签统计）

**前端实现**:
- 工作完成后显示评价提醒
- 评价表单支持星级、标签、文字、照片
- 企业详情页显示评价列表和统计

**测试用例**: 8个

## 数据流和错误处理

### 关键数据流

```
报名流程：
临工报名 → 检查时间冲突 → 检查认证状态 → 创建应用(pending)
         → 发送通知给企业 → 返回成功

企业审核流程：
企业接受 → 更新应用状态(accepted) → 发送通知给临工
企业拒绝 → 更新应用状态(rejected) → 发送通知给临工

主管选择流程：
选择主管 → 验证资格 → 更新应用状态(confirmed)
        → 发送通知给临工 → 返回成功

取消报名流程：
临工取消 → 检查状态 → 计算惩罚 → 更新应用状态(cancelled)
        → 扣信用分 → 发送通知给企业 → 返回成功
```

### 错误处理

```
时间冲突：
- 返回 400 + 冲突信息
- 前端显示冲突的工作列表

取消报名失败：
- 返回 400 + 原因（状态不对、时间限制等）
- 前端显示错误提示

主管选择失败：
- 返回 400 + 原因（资格不符、状态不对等）
- 前端显示错误提示

通知发送失败：
- 记录日志，不影响主流程
- 定时重试机制
```

## 总结

本设计方案通过渐进式实现，逐步完善临工招工流程：

1. **Phase 1** 确保核心业务逻辑完整（防冲突、支持取消、状态同步）
2. **Phase 2** 优化用户体验（UI调整、主管管理、通知提醒）
3. **Phase 3** 增强平台功能（评价系统、纠纷处理、数据分析）

每个阶段都包含完整的测试覆盖，确保质量。建议按优先级逐步实现，确保核心功能稳定后再添加高级功能。
