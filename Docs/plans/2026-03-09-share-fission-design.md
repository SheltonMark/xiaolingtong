# 小灵通 — 「帮他找」裂变拉新方案

> 日期：2026-03-09
> 目标：零预算，让用户成为天然传播媒介，快速获取第一批用户

---

## 一、核心洞察

### 1.1 用户已有的行为习惯

这些人每天都在微信群里做同样的事：

- 外贸老板在群里喊："谁家有304不锈钢管？急"
- 包工头在群里发："明天需要20个焊工，日薪350"
- 临工在群里问："哪里有活干？最好在松江附近"

**他们已经在传播了，只是载体是微信群里的文字消息。**

### 1.2 平台要做的事

不是"激励用户分享"，而是**让小灵通的分享卡片比打字更好用**。用户自然会用分享卡片替代打字，裂变就自然发生了。

### 1.3 关键前提

- 灵豆是平台唯一的信息解锁货币，不可提现
- **未解锁联系方式的用户，不能使用在线聊天功能** — 这让灵豆成为刚需
- 新用户注册送少量灵豆，降低首次体验门槛

---

## 二、「帮他找」机制

### 2.1 什么是「帮他找」

将现有的「分享」按钮替换为「帮他找」按钮。用户看到一条信息，自己用不上但认识合适的人，点「帮他找」转发到微信群或好友。

**与普通分享的本质区别：** 心理定位从"我在推广一个App"变成"我在帮朋友一个忙"。

### 2.2 两种角色，不同话术

| 转发者身份 | 场景 | 分享卡片标题示例 |
|-----------|------|----------------|
| 发布者本人 | 我发了招工信息，自己转发到群里找人 | 「急招焊工20人｜日薪350｜包吃住」 |
| 其他用户 | 我看到一条信息，帮忙转发给可能需要的人 | 「朋友急招焊工20人｜日薪350｜帮忙转一下」 |
| 发布者本人 | 我发了采购需求，自己转发找货源 | 「求购304不锈钢管500吨｜长期合作」 |
| 其他用户 | 我看到采购需求，帮忙转发给有货的人 | 「有人在找304不锈钢管｜你有货吗？」 |

系统自动判断：转发者 = 发布者 → 用发布者话术；转发者 ≠ 发布者 → 用「帮找」话术。

### 2.3 裂变链路

```
发布者A 发布信息
    ↓
A 自己转发到微信群（找人/找货）
    ↓
用户B 在群里看到，点进来浏览 ← 落地页展示完整信息
    ↓
B 想联系A → 需注册 + 花灵豆解锁 → B 成为新用户
    ↓
B 觉得这个平台有用，自己也开始发信息
    ↓
B 看到别的信息，帮忙转发给自己的圈子 → 循环
```

**每个环节都是用户的自发行为，零成本。**

## 三、新用户转化设计

### 3.1 落地页体验（未注册用户看到的）

新用户通过分享卡片进入时：

1. **信息详情完全可见** — 标题、描述、薪资、要求、图片全部展示，不做任何遮挡
2. **联系方式模糊处理** — 电话号码显示为 `138****5678`，微信号显示为 `wx****`
3. **底部强引导** — 「注册即送 3 灵豆，立即查看完整联系方式」
4. **在线聊天按钮置灰** — 提示「解锁联系方式后可在线沟通」

**设计原则：让用户先看到足够多的有价值信息，产生"我想联系这个人"的欲望，再引导注册。**

### 3.2 新人灵豆礼包

| 行为 | 获得灵豆 | 说明 |
|------|---------|------|
| 注册并选择角色 | 3 灵豆 | 够解锁 1 个联系方式，立刻体验核心功能 |
| 完成实名认证 | 5 灵豆 | 鼓励认证，提升平台信任度 |
| 首次发布信息 | 2 灵豆 | 鼓励从消费者变成内容贡献者 |

总计 10 灵豆，够解锁 3-5 个联系方式（取决于单次解锁定价），足以让新用户充分体验平台价值。

### 3.3 灵豆消耗后的自然付费转化

新人灵豆用完后，用户面临选择：
- 充值灵豆（付费转化）
- 通过「帮他找」转发赚灵豆（裂变循环）

两条路都对平台有利。

## 四、「帮他找」的灵豆激励

### 4.1 奖励规则

| 事件 | 转发者获得 | 条件 |
|------|-----------|------|
| 通过你的链接有新用户注册 | 2 灵豆 | 新用户完成选角色 |
| 通过你的链接有用户解锁了该条信息的联系方式 | 1 灵豆 | 不论新老用户 |

**发布者本人转发不获得灵豆奖励** — 发布者的动力是让更多人看到自己的信息，不需要额外激励。

### 4.2 防刷机制

- 每人每天通过「帮找」最多获得 20 灵豆
- 同一条信息最多奖励转发者 5 灵豆（防止同一条信息反复刷）
- 新用户注册奖励以设备维度去重，同一设备只算一次

### 4.3 为什么这个激励有效

灵豆虽然不能提现，但在「联系方式解锁 + 在线聊天」都锁在灵豆后面的前提下，灵豆 = 做生意的门票。对于活跃用户来说，每天都需要解锁新的联系方式，灵豆是实实在在的刚需。

「帮别人转发一下，自己省了充值的钱」— 这个动力对高频用户足够。

## 五、充值返佣（企业用户专属）

### 5.1 机制说明

灵豆激励解决的是"省钱"的动力，但对于有商业嗅觉的企业用户来说，**能赚钱才是真正的驱动力**。

充值返佣机制：企业用户 A 邀请了企业用户 B，B 每次充值灵豆，A 自动获得一定比例的现金返佣，返佣进入 A 的余额，余额可提现。

**为什么只限企业用户：** 临工端没有充值灵豆的场景（临工主要是找活、赚钱），充值行为集中在企业端（解锁联系方式、推广信息）。

### 5.2 返佣规则

| 项目 | 规则 |
|------|------|
| 返佣比例 | 后台统一配置，运营可随时调整（初始值 **10%**） |
| 返佣周期 | 持续返佣，B 的每一笔充值都返 |
| 到账时间 | B 充值成功后实时到账 A 的余额 |
| 提现门槛 | 余额满 **10 元** 可提现 |
| 提现方式 | 微信零钱（复用现有 Wallet 提现通道） |

**举例：**
- A 邀请了 5 个企业用户
- 这 5 个人每月各充值 100 元灵豆
- A 每月自动获得 5 × 100 × 10% = **50 元** 返佣
- 运营觉得比例高了，后台改成 8%，立即生效

### 5.3 为什么平台亏不了

灵豆是虚拟货币，充值收入是纯利润。分出 10% 给邀请人，平台仍然拿 90%。而这个用户是 A 带来的，没有 A 的邀请，平台连这 90% 都没有。

**本质上是用未来收入的一小部分，换取零成本获客。**

### 5.4 防刷与风控

| 风险 | 应对 |
|------|------|
| A 和 B 串通，B 充值后 A 提现 | 返佣比例只有 10%，串通无利可图（充100只返10） |
| A 用多个小号自充自返 | 同一设备/IP 注册的账号不计返佣；提现需实名认证 |
| B 充值后退款 | 退款时扣回对应返佣；若余额不足则挂账，下次返佣时抵扣 |

### 5.5 用户感知

A 在「我的」页面能看到：
- **我的余额**：累计返佣金额、可提现金额
- **我的团队**：邀请了多少企业用户、每人累计充值、每人累计返佣
- 每次有返佣到账时，收到系统通知：「您邀请的用户 XX 充值了 100 元，您获得 10 元返佣」

## 六、分享卡片设计

### 6.1 卡片样式要求

微信小程序分享卡片是用户的第一触点，决定了点击率。

**原则：像朋友发的消息，不像广告。**

- 标题：直接写核心信息（品名、数量、薪资），不写"小灵通平台"
- 图片：如果信息有图片，用第一张图；没有图片，用信息类型对应的默认图
- 描述：补充地点、时间等关键信息

### 6.2 各场景卡片标题模板

**招工信息：**
- 发布者转发：`{职位名称}｜日薪{金额}｜{地点}`
- 帮他找转发：`朋友在招{职位名称}｜日薪{金额}｜能去吗？`

**采购需求：**
- 发布者转发：`求购{品名}{数量}｜{规格要求}`
- 帮他找转发：`有人在找{品名}｜你有货吗？`

**工厂库存：**
- 发布者转发：`现货{品名}{数量}｜{价格}`
- 帮他找转发：`朋友有{品名}现货｜需要的看看`

**曝光信息：**
- 统一话术：`{公司名}被曝光｜小灵通曝光榜`

## 七、数据追踪

### 7.1 追踪参数

所有分享链接自动携带两个参数：
- `inviteCode` — 转发者的邀请码，用于追踪拉新归因
- `from=help` — 标记来源为「帮他找」（区分于其他入口）

### 7.2 关键指标

| 指标 | 说明 | 数据来源 |
|------|------|---------|
| 分享次数 | 每天有多少次「帮他找」转发 | 前端埋点 |
| 落地页访问量 | 多少人点击了分享卡片 | 页面 PV |
| 注册转化率 | 点击 → 注册的比例 | invite_records 表 |
| 解锁转化率 | 注册 → 解锁联系方式的比例 | contact_unlock 表 |
| 付费转化率 | 免费灵豆用完 → 充值的比例 | bean_orders 表 |
| 高价值转发者 | 谁带来的新用户最多 | invite_records 按 inviterId 聚合 |

首期不做管理后台看板，但数据库设计已支持后续统计。

## 八、实施优先级

### 第一期（MVP）

1. 「分享」按钮改为「帮他找」，分享卡片根据转发者身份显示不同话术
2. 分享链接携带 inviteCode 参数
3. 新用户注册送 3 灵豆
4. 未解锁联系方式时锁住在线聊天功能
5. 邀请关系记录（数据追踪，为后期运营提供依据）
6. 企业用户充值返佣机制（10% 持续返佣，余额可提现）

### 第二期（验证 MVP 后）

7. 「帮他找」灵豆激励（拉新奖励 + 解锁佣金）
8. 「我的邀请」页面 + 「我的团队」页面
9. 完成认证送灵豆、首次发布送灵豆
10. 分享卡片图片优化

### 第三期（用户量起来后）

11. 运营看板（管理后台统计裂变数据）
12. 高价值转发者识别和运营

## 九、技术可行性结论

**可行，改动量可控。**

- 现有灵豆体系（BeanTransaction）完全可复用，只需扩展 type 枚举
- 现有分享功能（onShareAppMessage）只需调整 title 生成逻辑和追加 inviteCode 参数
- User 表加两个 nullable 字段（inviteCode、invitedBy），不影响现有数据
- 新增一张 invite_records 表记录邀请关系
- 聊天功能加锁只需在发起聊天时校验是否已解锁对方联系方式
- 充值返佣复用现有 Wallet + WalletTransaction 体系，企业用户开通钱包，新增 transaction type `commission` 即可
- 返佣触发点嵌入现有支付回调 `handleBeanPay()`，充值成功时查询邀请人并写入返佣记录
- 提现复用现有提现逻辑（微信商家转账到零钱），需商户号满90天
- 所有改动都是追加性质，不修改现有业务逻辑
