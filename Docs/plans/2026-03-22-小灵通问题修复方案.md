# 小灵通问题修复方案

## 背景

本轮问题来自 `小灵通问题汇总.txt`，集中在招工管理、签到管理、工作记录、异常记录和导航链路。问题不是孤立缺陷，而是由状态流预取、实名显示口径、定位校验和页面职责过重共同造成。

## 总体判断

### 1. 状态流问题

- 招工管理点击“立即处理”进入的是报名管理页，但页面初始化时会同时预取结算详情接口。
- 当结算单尚未生成时，后端直接返回 `结算单不存在`，前端请求层会统一 toast，形成误报。
- 签到接口本身已具备“当天重复签到直接返回已有记录”的幂等行为，但签到页没有把“今日已签到”作为前端禁用态返回，导致刷新后看起来还能继续点。

### 2. 姓名显示口径不一致

- 报名管理页已经接入 `worker_cert.realName` 作为兜底。
- 签到名单、工时记录、考勤汇总、结算明细仍主要依赖 `users.nickname` / `users.name`。
- 同一临工在不同页面显示不同名字，造成“真实姓名未显示”。

### 3. 定位链路不闭环

- 签到页距离计算只在前端完成，后端没有二次校验。
- 当用户当前位置或岗位坐标缺失时，前端距离可能保持 `0` 或不可感知，导致页面显示与实际校验不一致。
- 招工详情页导航在岗位坐标缺失时使用了默认坐标，导航页又把获取用户当前位置当成展示前提，最终表现为“获取位置失败”。

### 4. 工作记录页职责过重

- 工作记录页同时承载现场拍照、工时记录、异常备注、签退和收工。
- 现场拍照已不再是用户当前关注点，但仍占据首屏和收工提示。
- 工时卡片信息密度偏高，扫读效率不高。

## 实施顺序

### 第一阶段：阻断误报和重复操作

1. 修复结算详情空态返回与流程页懒加载。
2. 修复签到页“今日已签到”禁用态。
3. 修复签到页和签到接口的定位校验。
4. 修复导航页的坐标缺失和无权限降级行为。

### 第二阶段：统一实名展示

1. 在工作模块统一实名解析逻辑。
2. 在结算模块统一实名解析逻辑。
3. 同步签到名单、工时页、考勤汇总、结算明细的显示来源。

### 第三阶段：收口交互与 UI

1. 异常记录默认时间改为当前时间。
2. 调整签到页拍照区顺序。
3. 移除工作记录页现场拍照模块。
4. 优化工时记录卡片排版。

## 验收口径

### 状态流

- 从“招工管理 -> 待处理 -> 立即处理”进入报名页，不再弹出“结算单不存在”。
- 已签到用户刷新签到页后，按钮为禁用态，文案明确为“今日已签到”。

### 实名

- 同一工人在报名管理、签到名单、工时记录、考勤汇总、结算明细中的显示姓名一致。
- 优先显示实名认证姓名，缺失时再回退到 `users.name` 或 `nickname`。

### 定位

- 岗位未配置坐标时，不允许发起签到，也不允许进入伪导航。
- 用户拒绝定位授权时，导航页仍可打开目标地点；签到页则明确提示需要定位。
- 前端超距和后端超距都能阻止签到。

### UI

- 签到页照片按拍摄顺序展示，拍照按钮始终在末尾。
- 工时记录页不再显示“现场拍照记录”卡片。
- 工时卡片首屏即可清楚看到姓名、签到、签退、工时和主要操作。

## 环境隔离要求

为避免本项目与其他并行项目相互污染，开发期间执行以下要求：

- 代码隔离：使用独立分支或独立 `git worktree`。
- 配置隔离：本项目独立 `.env`、接口地址、微信配置、对象存储配置。
- 服务隔离：MySQL、Redis、Node 端口、容器名、卷名独立。
- 数据隔离：开发库、测试库与其他项目完全分离。
- 文件隔离：日志、上传目录、缓存目录增加项目标识。
- 提交流程隔离：本轮修复全部通过自测后再合并。
