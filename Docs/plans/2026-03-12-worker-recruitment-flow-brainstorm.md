## 需求1: 临工端报名时间冲突检查

### 问题描述
临工报名多个工作时，如果时间冲突则不允许报名。

### 当前状态
✅ 已在设计文档中提及（4.3 报名流程）

### 实现方案

**后端逻辑**:
```typescript
// 检查时间冲突
async checkTimeConflict(workerId: number, jobId: number) {
  const job = await this.jobRepo.findOne({ where: { id: jobId } });

  // 查询该工人已确认的应用
  const confirmedApps = await this.appRepo.find({
    where: {
      workerId,
      status: In(['accepted', 'confirmed', 'working'])
    },
    relations: ['job']
  });

  // 检查时间是否重叠
  for (const app of confirmedApps) {
    if (this.hasTimeOverlap(job, app.job)) {
      throw new BadRequestException('工作时间冲突，无法报名');
    }
  }
}

// 时间重叠判断
private hasTimeOverlap(job1: Job, job2: Job): boolean {
  const start1 = new Date(job1.dateStart);
  const end1 = new Date(job1.dateEnd);
  const start2 = new Date(job2.dateStart);
  const end2 = new Date(job2.dateEnd);

  return !(end1 < start2 || end2 < start1);
}
```

**前端提示**:
- 报名时检查是否有时间冲突
- 冲突时显示已报名的工作信息
- 提示用户取消或修改已报名工作

## 需求2: 临工端可以取消报名

### 问题描述
临工需要能够取消已报名的工作。

### 实现方案

**API 端点**:
```
DELETE /api/applications/:applicationId/cancel
```

**业务规则**:
```
允许取消的状态:
- pending (待审核) ✅
- accepted (已接受) ✅
- confirmed (已确认) ⚠️ (需要扣信用分)

不允许取消的状态:
- working (进行中) ❌
- done (已完成) ❌
- rejected (已拒绝) ❌
- released (已释放) ❌
- cancelled (已取消) ❌
```

**惩罚机制**:
```
取消时间点 | 惩罚
-----------|------
报名后 → 工作前24小时 | 无惩罚
工作前24小时 → 工作前12小时 | 扣信用分5分
工作前12小时 → 工作开始 | 扣信用分10分 + 限制报名24小时
工作开始后 | 扣信用分20分 + 限制报名7天
```

**前端界面**:
```
我的报名 → 选择应用 → [取消报名]
  ↓
确认取消对话框
  ├─ 显示取消时间和可能的惩罚
  ├─ 显示已报名的其他工作
  └─ [确认取消] [返回]
```

### 问题描述
确保临工端看到的状态和企业端看到的状态一致。

### 状态映射表

```
后端状态 | 临工端显示 | 企业端显示 | 说明
---------|----------|----------|------
pending | 待审核 | 待审核 | 企业还未审核
accepted | 已入选 | 已接受 | 企业已接受，待确认
confirmed | 进行中 | 已确认 | 企业已确认，临工已确认出勤
working | 进行中 | 进行中 | 工作正在进行
done | 已完成 | 已完成 | 工作已完成，待结算
rejected | 未通过 | 已拒绝 | 企业拒绝了应用
released | 已释放 | 已释放 | 临工未及时确认，名额被释放
cancelled | 已取消 | 已取消 | 临工主动取消或企业取消
```

## 需求5: 企业端和临工端状态双向匹配

### 问题描述
确保企业端看到的招工状态和临工端看到的应用状态完全对应。

### 状态流转图

```
企业视角                    临工视角
─────────────────────────────────────

发布招工 (recruiting)
    ↓
临工报名 ──────────────→ 待审核 (pending)
    ↓
企业审核
├─ 接受 ──────────────→ 已入选 (accepted)
│   ↓
│   企业选择主管
│   ├─ 选择成功 ────→ 进行中 (confirmed)
│   │   ↓
│   │   工作开始 ────→ 进行中 (working)
│   │   ↓
│   │   工作完成 ────→ 已完成 (done)
│   │   ↓
│   │   结算完成 ────→ 已完成 (done)
│   │
│   └─ 未及时确认 ──→ 已释放 (released)
│
└─ 拒绝 ──────────────→ 未通过 (rejected)

企业取消 ──────────────→ 已取消 (cancelled)
临工取消 ──────────────→ 已取消 (cancelled)
```

### 实现方案

**后端状态转换验证**:
```typescript
// 有效的状态转换
const VALID_TRANSITIONS = {
  pending: ['accepted', 'rejected', 'cancelled'],
  accepted: ['confirmed', 'rejected', 'released', 'cancelled'],
  confirmed: ['working', 'released', 'cancelled'],
  working: ['done', 'cancelled'],
  done: [],
  rejected: [],
  released: [],
  cancelled: []
};

// 验证状态转换
validateTransition(currentStatus: string, newStatus: string): boolean {
  return VALID_TRANSITIONS[currentStatus]?.includes(newStatus) || false;
}
```

## 需求6: 企业端选择主管并管理考勤

### 问题描述
企业确认招聘零工后，从零工中选择一个主管，对临工的考勤和完成度进行管理。

### 主管选择流程

```
企业查看应用列表
    ↓
选择"已接受"的应用
    ↓
点击"选择主管"
    ↓
显示符合条件的临工列表
├─ 信用分 >= 95
├─ 累计接单 >= 10
└─ 无违规记录
    ↓
选择一个临工作为主管
    ↓
确认选择
    ↓
系统更新状态为 'confirmed'
推送通知给主管
```

### 主管权限和职责

**主管权限**:
```
1. 考勤管理
   - 查看所有临工的签到情况
   - 标记临工为已签到/未签到
   - 确认开工人数

2. 工时/计件记录
   - 记录每个临工的工时
   - 记录每个临工的计件数
   - 上传现场照片

3. 异常上报
   - 上报临工旷工
   - 上报临工早退
   - 上报工时造假
   - 上报其他异常

4. 结算确认
   - 查看工时/计件汇总
   - 确认结算单
   - 提交给企业支付
```

**主管收益**:
```
主管收益 = 正常工资 + 监管服务费

监管服务费 = 监管人数 × 单价 × 工时/件数

示例:
- 监管10人，每人8小时，单价5元/小时
- 监管服务费 = 10 × 5 × 8 = 400元
```
