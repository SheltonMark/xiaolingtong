# 用工流程与工作结算页整合方案

更新时间：2026-03-22
适用分支：`fix/issue-batch-20260322`

## 1. 改造目标

当前 [job-process](C:/Users/15700/Desktop/work/project/小灵通/.worktrees/fix-issue-batch-20260322/pages/job-process/job-process.wxml) 和 [settlement](C:/Users/15700/Desktop/work/project/小灵通/.worktrees/fix-issue-batch-20260322/pages/settlement/settlement.wxml) 都在承载“考勤汇总 + 结算明细”。

结果是：

- 用户层面：两个页面语义重叠，不清楚该从哪里处理用工单。
- 开发层面：同一套数据请求、动作按钮、状态文案在两页重复维护。
- 回归层面：修一页容易漏一页，线上行为不一致。

本次整合目标：

- 只保留一个真正承载完整流程的主页面。
- 另一个页面只保留兼容跳转能力，不再维护完整业务 UI。
- 通知、列表页、待办入口统一跳到同一个主页面，通过 `tab` 决定落点。

## 2. 页面定位

### 2.1 主页面

保留 [job-process](C:/Users/15700/Desktop/work/project/小灵通/.worktrees/fix-issue-batch-20260322/pages/job-process/job-process.wxml) 作为唯一主页面。

建议对外名称：

- 企业端：`用工工作台`
- 工人端：`结算详情`
- 主管端：`考勤与结算`

页面职责：

- 企业端承载三段流程：`applications`、`attendance`、`settlement`
- 工人端默认只看 `settlement`
- 主管端默认看 `attendance`，必要时切 `settlement`

### 2.2 兼容页面

保留 [settlement](C:/Users/15700/Desktop/work/project/小灵通/.worktrees/fix-issue-batch-20260322/pages/settlement/settlement.wxml) 作为兼容壳页。

页面职责：

- 接收旧通知链接
- 接收旧页面跳转
- 接收历史分享路径
- 进入后立即跳转到 `job-process`

不再承担：

- 独立渲染结算 UI
- 独立加载考勤和结算数据
- 独立处理支付、确认、争议等动作

## 3. 路由方案

## 3.1 统一目标路由

统一目标路由为：

```text
/pages/job-process/job-process?jobId={jobId}&tab={tab}&role={role}&viewOnly={0|1}
```

建议约定：

- `tab=applications`：企业端报名管理
- `tab=attendance`：主管或企业查看考勤
- `tab=settlement`：企业、工人查看结算
- `role=enterprise|worker|manager`
- `viewOnly=1`：只读进入，例如通知跳转、工人视角

## 3.2 旧路由兼容

旧入口仍允许访问：

```text
/pages/settlement/settlement?jobId={jobId}&role={role}
```

但 [settlement.js](C:/Users/15700/Desktop/work/project/小灵通/.worktrees/fix-issue-batch-20260322/pages/settlement/settlement.js) 只做参数转换并跳转：

- enterprise -> `tab=settlement`
- worker -> `tab=settlement&viewOnly=1`
- manager -> `tab=attendance`

推荐实现：

```js
wx.redirectTo({
  url: `/pages/job-process/job-process?jobId=${jobId}&tab=${targetTab}&role=${role}&viewOnly=${viewOnly}`
})
```

## 3.3 各入口映射

入口统一映射如下：

- 我的发布 -> `job-process?tab=applications`
- 报名处理通知 -> `job-process?tab=applications`
- 考勤待确认 -> `job-process?tab=attendance`
- 结算待支付 -> `job-process?tab=settlement`
- 工人结算通知 -> `job-process?tab=settlement&role=worker&viewOnly=1`
- 旧 settlement 页跳转 -> 由 settlement 壳页转发到 `job-process`

## 4. 页面结构方案

主页面 [job-process](C:/Users/15700/Desktop/work/project/小灵通/.worktrees/fix-issue-batch-20260322/pages/job-process/job-process.wxml) 建议统一成：

### 4.1 顶部统一头部

只保留一份，不在不同 tab 重复展示。

头部信息：

- 岗位标题
- 企业名称
- 日期范围
- 当前阶段标签
- 关键状态提示

### 4.2 Tab 分区

企业端：

- 报名管理
- 考勤汇总
- 结算明细

工人端：

- 不展示 tabs
- 默认进入 `settlement`

主管端：

- 仅展示 `attendance`、`settlement`

### 4.3 Tab 内内容边界

`applications` 只放：

- 报名概览
- 报名筛选
- 报名列表
- 录用/拒绝/设主管动作

`attendance` 只放：

- 考勤概览
- 考勤明细表
- 现场照片
- 汇总单状态
- 确认汇总单/发起争议动作

`settlement` 只放：

- 结算卡片
- 流程状态
- 人员结算明细
- 费用拆分
- 支付/确认工时/评价动作

原则：

- 考勤 tab 不再重复显示完整结算卡片
- 结算 tab 不再重复显示完整考勤汇总表
- 只通过“状态摘要”串联，不做整块复制

## 6. 最小改动实施步骤

按“先消重、再抽组件、最后清理旧页”的顺序做。

### 第 1 步：先收口路由

目标：

- 先统一用户入口，不先大拆模板。

改动：

- 改 [settlement.js](C:/Users/15700/Desktop/work/project/小灵通/.worktrees/fix-issue-batch-20260322/pages/settlement/settlement.js)
- `onLoad` 中不再加载数据
- 直接按参数 `redirectTo` 到 `job-process`

收益：

- 立刻消除“双页面心智”
- 后续新逻辑只会落在一个页面

### 第 2 步：`job-process` 成为唯一数据页

目标：

- 所有业务数据只在 `job-process.js` 里请求

改动：

- 保留 `loadManage`
- 保留 `loadAttendance`
- 保留 `loadSettlement`
- 保留 `onConfirmAttendance`
- 保留 `onPay`
- 保留 `onConfirmSettlement`
- 将 settlement 页对应逻辑从主业务链上移除

收益：

- 避免两页重复请求、重复状态维护

### 第 3 步：抽出两个重复块

最小优先顺序：

1. `attendance-panel`
2. `settlement-panel`

原因：

- 这两块正好是 `job-process` 和 `settlement` 之前最重复的部分
- 抽完后模板会立刻瘦下来

### 第 4 步：抽统一头部

改动：

- 新增 `workflow-header`
- `job-process` 只保留一份顶部头部

收益：

- 不同 tab 的信息框口径一致
- 后续做状态样式时只改一处

### 第 5 步：清空 settlement 页面 UI

改动：

- [settlement.wxml](C:/Users/15700/Desktop/work/project/小灵通/.worktrees/fix-issue-batch-20260322/pages/settlement/settlement.wxml) 改成极简占位
- [settlement.js](C:/Users/15700/Desktop/work/project/小灵通/.worktrees/fix-issue-batch-20260322/pages/settlement/settlement.js) 只保留转发逻辑

占位文案建议：

- “正在跳转到用工工作台…”

### 第 6 步：统一通知跳转

改动：

- 通知 link 统一指向 `job-process`
- 不再新增指向 `settlement` 的新链接

收益：

- 线上入口逐步收口
- settlement 壳页只保留历史兼容

## 8. 验收标准

### 8.1 用户体验

- 企业只会感知到一个主页面在处理整张用工单
- 工人只会感知到一个结算详情入口
- 通知点击后不再出现“同一业务跳到不同页面”的割裂感

### 8.2 开发维护

- 考勤相关请求只在一处维护
- 结算相关请求只在一处维护
- 支付、确认工时、确认考勤动作只在一处绑定

### 8.3 回归范围

必须覆盖：

- 企业从我的发布进入
- 企业从通知进入
- 工人从通知进入
- 主管从待办进入
- settlement 历史链接进入后能正确跳转

## 9. 风险与注意点

- 不能直接删除 `settlement` 页面，否则历史通知和旧链接会失效。
- `redirectTo` 层级有限，如果来源页层级复杂，可根据实际情况退回 `reLaunch` 或 `navigateTo`。
- `job-process` 现在职责会变重，短期先接受，等稳定后再考虑聚合接口。
- 如果后端后续提供 `/jobs/:id/workflow-detail` 聚合接口，可以再做第二轮前端收敛。

## 10. 推荐结论

本次最稳妥的做法不是“继续分别优化两个页面”，而是：

- 保留 `job-process` 作为唯一主页面
- 将 `settlement` 降级为兼容跳转页
- 用 3 个面板组件拆掉重复内容
- 用统一路由参数控制不同角色和落点

这是最小改动、收益最大、风险最低的整合方式。
